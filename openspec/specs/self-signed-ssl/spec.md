## ADDED Requirements

### Requirement: Directory init
`init()` creates the OpenSSL CA directory structure for managing self-signed certificates.

#### Scenario: Creates CA directory skeleton
- **WHEN** `init()` is called with a base directory path
- **THEN** it creates the following subdirectory structure:
  - `certs/` -- issued certificates
  - `private/` -- private keys (restricted permissions)
  - `newcerts/` -- new certificate storage
  - `crl/` -- certificate revocation list storage
  - `csr/` -- certificate signing requests

#### Scenario: Creates required index and serial files
- **WHEN** `init()` creates the directory structure
- **THEN** it creates an empty `index.txt` file for tracking issued certs
- **AND** creates a `serial` file initialized with `01` as the starting serial number

#### Scenario: Does not overwrite existing structure
- **WHEN** `init()` is called on a directory that already has the CA structure
- **THEN** the function returns early without destroying existing files

### Requirement: Root CA
`generate_root()` generates a 20-year self-signed Root CA certificate using RSA 2048-bit key.

#### Scenario: Generates Root CA key and certificate
- **WHEN** `generate_root()` is called
- **THEN** it generates an RSA 2048-bit private key
- **AND** creates a self-signed Root CA certificate with 20-year validity
- **AND** stores the key in `private/ca.key.pem` and the cert in `certs/ca.cert.pem`

#### Scenario: Root CA distinguished name is configurable
- **WHEN** `generate_root()` is called
- **THEN** the certificate subject includes configurable fields: Country (C), State (ST), Organization (O), and Common Name (CN)

### Requirement: Domain certificate
`generate_cert()` issues wildcard certificates with Subject Alternative Names (SAN) for multiple domains.

#### Scenario: Issues wildcard certificate
- **WHEN** `generate_cert()` is called with a domain name (e.g., `example.com`)
- **THEN** it generates a certificate with both `example.com` and `*.example.com` in the SAN list

#### Scenario: Supports multiple domain SANs
- **WHEN** `generate_cert()` is called with a list of additional domain names
- **THEN** each domain is added to the SAN extension of the certificate

#### Scenario: Signs with Root CA
- **WHEN** `generate_cert()` creates a certificate
- **THEN** it is signed by the Root CA generated by `generate_root()`

### Requirement: Certificate chain
Outputs a fullchain bundle file containing both the domain certificate and the Root CA certificate for use in nginx.

#### Scenario: Creates fullchain bundle
- **WHEN** a domain certificate is issued
- **THEN** a `fullchain.pem` file is created
- **AND** it contains the domain certificate followed by the Root CA certificate

#### Scenario: Fullchain is ready for nginx
- **WHEN** the fullchain bundle is generated
- **THEN** its format is compatible with the `ssl_certificate` directive in nginx

### Requirement: Version management
Each certificate issuance creates a timestamped version directory to preserve history.

#### Scenario: Creates versioned directory for each issuance
- **WHEN** `generate_cert()` issues a certificate for a domain
- **THEN** certificates and keys are stored in a directory named with the issuance timestamp (e.g., `example.com/2025-01-15-143022/`)

#### Scenario: Preserves previous certificates
- **WHEN** a new certificate is issued for a domain that already has certificates
- **THEN** the previous version remains intact in its own timestamped directory

### Requirement: Symlinks
The latest certificates are linked via symlinks in the domain directory for easy reference.

#### Scenario: Creates latest symlinks
- **WHEN** a new certificate version is created
- **THEN** symlinks named `cert.pem`, `fullchain.pem`, and `privkey.pem` are created (or updated) in the domain root directory, pointing to the latest version

#### Scenario: Symlinks are updated on re-issuance
- **WHEN** a subsequent certificate is issued for the same domain
- **THEN** the existing symlinks are updated to point to the new version's files
